Spectra Logic Internship

May 2017 - Aug 2017, Feb 2018 - May 2018

I interned at Spectra Logic as an Electrical Engineering Intern full time for the summer of 2017 and part time during the spring 2018 semester. Over the summer, I had three main projects.

The first project was modeling current, speed, and position control of a BLDC motor in Simulink. The first BLDC motor model used a combination of circuit elements and signal blocks, so the modeling of the BLDC motor itself was more accurate, but because a combination of element types was used, simulations took a very long time to run. The second model aimed to decrease the run time of the simulation by using only signal blocks based on the mathematical model of a BLDC motor. The second model acted similarly enough to the circuit/signal combination model during the motor speeds of interest, so I used this second, faster BLDC model to design a PID controller that acted on the BLDC motor model. I first started with controlling current in the motor, then added speed and position control.

For my second project, I designed and executed tests to determine optimum operational parameters of an optical sensor for use in a tape storage library. The optical sensor was used to check that the position of a robot within the library was correct, as well as allow fine-tuning of the robot position, in order to avoid the robot jamming or damaging tapes. The tests I designed were to determine the optimal angle of the robot actuator, optimal distance from the target, and optimal position in the vertical and horizontal space within the library. I analyzed the resulting data using Matlab to create 3D plots of intensity vs. distance & angle and intensity vs. vertical & horizontal position, which clearly showed optimal operating conditions (or ranges of acceptable conditions) of the optical sensor on the robot.

My final project was to design and build a test setup that could replicate and catch a startup corruption error of a robotic system. To achieve this goal, I built a MOSFET-based reset circuit that would trigger the power reset of the system under test, and used an Arduino to control this circuit. I programmed the Arduino to trigger the reset on a variable time delay (as timing was believed to be the cause of the corruption error) and check over the serial terminal for corruption. "Corruption" was detected by checking for certain phrases in the serial output that were outputted during normal startup and failed startup conditions.
